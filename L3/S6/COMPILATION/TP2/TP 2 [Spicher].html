TP 2

Point de départ

L'objectif du TP est de construire un projet complet permettant d'analyser lexicalement et syntaxiquement un fichier d'entrée et pour finalement l'évaluer. Afin de réaliser cet objectif un squelette de projet vous est proposé.

  • tp2.zip

Après avoir téléchargé et décompressé ce fichier vous obtenez les fichiers suivants constituant le projet :

  • makefile : un fichier de spécification pour l'outil make
  • ast.ml : le fichier source du module Ast s'intéressant à la syntaxe
  • ast.mli : le fichier interface (correspondant au header en C) du module Ast
  • parser.mly : le fichier de spécification de l'analyseur syntaxique
  • lexer.mll : le fichier de spécification de l'analyseur lexical
  • main.ml : le fichier source contenant le code du programme principal

Makefile et organisation du code

Le fichier makefile est constitué de deux parties :

  1. Une partie customisable permettant de spécifier le nom de l'exécutable et la liste des fichiers constituant le projet ; dans notre cas :
    # Nom du fichier exécutable généré
EXEC=tp2
 
# Liste des sources du projet, données dans l'ordre
SOURCES= \
	ast.mli     \
	ast.ml      \
	parser.mly  \
	lexer.mll   \
	main.ml
  2. Une partie générique à ne pas éditer permettant de spécifier les règles de compilation

La compilation du projet simplement à l'aide de la commande make : 

$ make
ocamlyacc -v parser.mly
ocamllex -o lexer.ml lexer.mll
11 states, 270 transitions, table size 1146 bytes
ocamldep  ast.mli ast.ml parser.ml lexer.ml main.ml > depend
ocamlc -o ast.cmi -c ast.mli
ocamlc -o ast.cmo -c ast.ml
ocamlc -o parser.cmo -c parser.ml
ocamlc -o lexer.cmo -c lexer.ml
ocamlc -o main.cmo -c main.ml
ocamlc -o tp2 ast.cmo parser.cmo lexer.cmo main.cmo

Pour supprimer l'ensemble des fichiers générés, on utilisera la commande make clean : 

$ make clean
rm -rf tp2 parser.ml parser.mli parser.output lexer.ml ast.cmi ast.cmo parser.cmo lexer.cmo main.cmo ast.cmi ast.cmi parser.cmi lexer.cmi main.cmi depend

La compilation du projet produit un fichier exécutable nommé dans notre cas tp2 (variable EXEC dans makefile), qu'il convient de lancer comme tout autre programme : 

$ ./tp2 
1 ;;
1 => 1
1 + 2 ;;
(1) + (2) => 3
1 + 2 + 3 ;;
((1) + (2)) + (3) => 6
Bye

Notre programme se présente comme une boucle interactive attendant en entrée standard une suite d'expressions arithmétiques terminées par un double point-virgule ;;. Chaque expression est analysée et évaluée. Le programme affiche l'arbre de syntaxe reconnu puis la valeur évaluée. Pour quitter le programme, il suffit d'indiquer la fin de fichier sur l'entrée standard, par un Ctrl+D par exemple. Le code correspondant se trouve dans main.ml.

Syntaxe abstraite

Les fichiers ast.ml et ast.mli contiennent la description de la syntaxe abstraite (Abstract Syntax Tree) et de son évaluation.

En OCaml, chaque fichier .ml (accompagné de son .mli) le cas échéant correspond à la spécification d'un module. Le nom du module est le nom du fichier sans l'extension .ml en commençant toujours par une majuscule. Dans notre cas, le module s'appelle Ast.

Le fichier .mli (correspondant OCaml du fichier .h en C) décrit l'interface du module à travers sa signature qui regroupe l'ensemble des types (nom du type et sa spécification si l'on souhaite la faire connaître hors du module) et valeurs déclarés (identifiant associé et type) qui seront autorisés d'utilisation dans d'autres modules. Dans notre cas : 

(* Syntaxe abstraite *)
type expr
 
(* Constructeurs d'expression *)
val cst: int -> expr
val add: expr -> expr -> expr
val neg: expr -> expr
 
(* Conversion en chaîne de caractères pour affichage *)
val string_of_expr: expr -> string
 
(* Evaluateur *)
val eval: expr -> int

Tout ajout de définition dans un fichier .ml devra donc être suivi d'une modification de son .mli uniquement si l'on souhaite faire connaître la nouvelle déclaration à l'extérieur du module. La présence d'un .mli n'est pas obligatoire ; en cas d'absence, OCaml en considère un par défaut dans lequel rien n'est caché.

  • Ajouter le constructeur correspondant à la multiplication au module Ast qui correspondra à l'ajout dans le .mli de la déclaration 
    val mul: expr -> expr -> expr

Précédences

Souvent, la grammaire spécifiée n'est pas LALR(1). C'est le cas de la nôtre. En effet, on peut observer l'état suivant dans parser.output : 

state 10
        expr : expr . PLUS expr  (4)
        expr : expr PLUS expr .  (4)

        .  reduce 4

Le conflit n'est pas indiqué par ocamlyacc car une précédence est spécifiée dans le fichier .mly : 

%left PLUS

qui spécifie que le token est associatif à gauche.

  • Modifier %left en %right et observer la modification de l'associativité associée à 1 + 2 + 3 ;;
  • Remettre finalement %left, l'associativité attendue
  • Décommenter la règle de la grammaire et observer l'apparition des conflits
  • Corriger les conflits en spécifiant que la multiplication est également associative à gauche et de priorité plus élevée que l'addition
  • Décommenter la règle de la grammaire (on notera l'utilisation d'un sucre syntaxique dans l'action sémantique) et observer l'apparition des conflits
  • Corriger les conflits en spécifiant que la soustraction est également associative à gauche et de même priorité que l'addition

  • Observer l'arbre associé à l'expression -1 * 2 ; convient-elle ? Modifier la précédence du moins unaire sur la multiplication ; nous rajouterons pour cela une précédence non-associative UMINUS ne correspondant à aucun token et préciserons sur la production correspondant à la négation la précédence à utiliser par %prec UMINUS ; observer l'arbre obtenu sur -1 * 2 ;;
